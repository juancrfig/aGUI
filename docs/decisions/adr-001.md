# aGUI Design Decisions

## Decision 1: MVP Scope — Plugin System (Option 2)

**Status:** Accepted

For the hackathon MVP, we stay within the Hermes plugin/theme system rather than forking the dashboard source. This is a proof of concept demonstrating the adaptive GUI vision. If well-received, we pursue Option 1 (full dashboard fork) post-hackathon.

## Decision 2: Seed Interface — Taskbar + Notification Trail

**Status:** Accepted

The default view is not empty. It contains:

1. **Taskbar** — Quick-access buttons for pre-defined common functionalities (cron, skills, config, etc.). These are hardcoded for MVP to demonstrate the concept, with the understanding that long-term these will emerge dynamically from session history.

2. **"+" Button** — Opens a chat/voice input where the user tells the agent what they want to do. The agent responds with a new animated button on the taskbar, guiding the user: "Hey! Click here!"

3. **Notification Trail** — A persistent notification inbox. Warnings, reminders, and general notifications appear as toasts first, then are saved here for later review.

## Decision 3: Taskbar Widgets — Link Buttons (Not Live Widgets)

**Status:** Accepted

For MVP, taskbar items are animated link buttons that navigate to existing dashboard tabs. They do not show live preview data. Post-MVP, these can be promoted to live widgets with backend polling.

## Decision 4: GitHub Issue Flow — Transparency First

**Status:** Accepted

When the user requests something not currently possible:

1. The agent attempts to fulfill the request using available tools
2. If impossible without source code modification, the agent clearly communicates: "This requires hacking the source code — expect things to not work right away"
3. The agent asks the user: report an issue, contribute a PR, or do nothing
4. If reporting: generate a GitHub issue draft with pre-filled title/body, show it to the user, and provide a link for quick publishing

This prevents agents from silently modifying code that benefits no one, and educates users about the boundary between "available" and "requires contribution."

## Decision 5: Widget Management — Rearrange, Close, Trash

**Status:** Accepted

MVP widget management features:
- Rearrange taskbar items (drag/drop or arrows)
- Close/remove items from taskbar
- Trash (recover deleted items)

Post-MVP: resize, persist layout across sessions, promote to live widgets.

## Decision 6: Bottom-Up Vision — Hybrid for MVP

**Status:** Accepted

The long-term vision is fully emergent: both user behavior and agent reasoning shape the UI dynamically, with session history driving what appears.

For MVP, we use pre-defined categories on the taskbar to demonstrate the concept. This is acknowledged as a top-down compromise for the sake of a working demo, with full emergence as the post-hackathon goal.
